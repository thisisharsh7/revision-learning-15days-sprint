## What is Regression Testing?

Imagine you have a recipe for chocolate cake. Every time you make it, it tastes good.

Now you want to make it better, so you add more chocolate. But what if that makes it taste worse? Or changes other things you didn't expect?

**Regression testing** = Making the cake again after changes to check it still tastes good AND nothing else broke.

**For Valhalla routing:**
- Old recipe = routes that work now
- Adding chocolate = changing the routing code
- Making cake again = running 10,000 test routes
- Checking taste = comparing old routes vs new routes

## How Valhalla Routing Works

1. You say: "I want to go from my house to the park"
2. Valhalla looks at the map (OpenStreetMap data)
3. It finds all possible roads
4. It picks the best path (fastest, shortest, or whatever you want)
5. It tells you: "Go straight, turn left, turn right, you're there!"

**The testing tool:**
- Tests if Valhalla still finds good paths after code changes
- Checks 10,000 different trips
- Shows what changed

## Virtual Scrolling (For 10,000 Routes)

**Problem:**
If you try to show 10,000 items in a list, the browser crashes. Too much!

**Solution:**
Only show what you can see on screen. Like looking through a window.

**Example:**
- You have 10,000 routes
- Your screen can show 20 routes at a time
- Virtual scrolling only renders those 20
- When you scroll, it shows the next 20
- Browser stays fast!

**Analogy:**
Imagine you have 10,000 photos. You don't print all of them at once! You only look at the ones in front of you. When you want to see more, you flip the page.

## GitHub Actions (CI/CD)

**What is it?**
Automatic robots that run code when you do something on GitHub.

**Example:**
1. You create a pull request (want to add code)
2. Robot wakes up automatically
3. Robot runs your code
4. Robot runs tests
5. Robot tells you if everything works

**For the routing project:**
- When someone makes a PR to Valhalla
- Robot runs 10,000 test routes automatically
- Robot compares results
- Robot saves results
- Maintainers can check results in the web app

**Analogy:**
It's like having a helper who checks your homework automatically every time you finish it.

## Route Geometry

**What is it?**
The actual path a route takes. The squiggly line on the map.

**Components:**
- A list of coordinates (lat, lon)
- Connected by lines
- Forms the route path

**Example:**
```
Start: (12.34, 56.78)
Point 2: (12.35, 56.79)
Point 3: (12.36, 56.80)
End: (12.37, 56.81)
```

Draw lines between them = your route!

**Why it matters:**
If route geometry changes, the path changed. Maybe good, maybe bad. Need to check!

## Comparing Two Routes

**How to know if routes are different?**

**Check these:**
1. **Distance** - Is it longer or shorter?
2. **Time** - Does it take more or less time?
3. **Path** - Does it go different streets?
4. **Instructions** - Are the turn-by-turn directions different?

**Example:**
```
Old route:  10 km, 15 min, "Go straight, turn left"
New route:  12 km, 18 min, "Go straight, turn right, turn left"

Different? YES!
How much? 2km longer, 3min slower, different instructions
```

## OpenStreetMap (OSM)

**What is it?**
Like Google Maps, but:
- Free and open
- Anyone can edit
- Anyone can use the data
- Community-made

**Why Valhalla uses it:**
Valhalla needs map data to find routes. OSM gives free map data!

**What's in OSM:**
- Roads (highways, streets, bike paths)
- Rules (one-way, no trucks allowed, etc.)
- Places (parks, buildings, etc.)

**For the testing tool:**
Test routes use OSM data. Need to test different road types from OSM.

## Costing Models (car, bike, pedestrian)

**What is it?**
Different ways to calculate "best" route.

**Car:**
- Uses highways
- Fast roads preferred
- Can't use bike-only paths

**Bicycle:**
- Uses bike lanes when possible
- Avoids highways
- Prefers flat terrain

**Pedestrian:**
- Uses sidewalks
- Can use stairs
- Shortest distance

**Analogy:**
It's like asking "what's the best way to get there?" but the answer depends on if you're driving, biking, or walking!

## Turn-by-Turn Instructions

**What is it?**
Step-by-step directions.

**Example:**
1. Head north on Main Street
2. Turn right onto Park Avenue
3. Turn left onto Oak Street
4. You have arrived!

**Why it matters for testing:**
If instructions change, maybe the route changed. Need to check if it's still good!

## What is "Code Coverage"?

**Definition:**
How much of the code gets tested.

**Analogy:**
Imagine Valhalla code is a big house with 100 rooms. If your tests only check 50 rooms, you have 50% coverage. You want to check all 100 rooms!

**For testing:**
- Create test routes that use different parts of Valhalla
- Test highways, bike paths, walking routes
- Test turn restrictions, one-ways, etc.
- Cover as many "rooms" (code paths) as possible

**Goal:**
Cover every single statement = Check every room!

## JSON Format (Data Structure)

**What is it?**
A way to organize data in text.

**Example:**
```json
{
  "route_id": "route_123",
  "distance": 10.5,
  "time": 900,
  "geometry": [[12.34, 56.78], [12.35, 56.79]]
}
```

**Analogy:**
It's like writing down information in an organized notebook so computers can read it.
